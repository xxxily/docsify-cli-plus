## Project Overview

docsify-cli-plus is an enhanced CLI tool for [Docsify](https://docsify.js.org/), a magical documentation site generator. This fork adds the `start` command for zero-configuration document serving.

### Key Features

- **init**: Generate new documentation projects with customizable templates
- **serve**: Run a development server with live-reload
- **start**: Zero-configuration server with auto-generated sidebar (NEW)
- **generate**: Auto-generate sidebar files from directory structure

### Technology Stack

- **Runtime**: Node.js >= 20.11.0
- **CLI Framework**: [yargs](https://github.com/yargs/yargs) with [yargonaut](https://github.com/nickmccurdy/yargonaut) for styling
- **Build Tool**: [Rollup](https://rollupjs.org/) for bundling
- **Server**: [connect](https://github.com/senchalabs/connect) with middleware
- **Live Reload**: [livereload](https://github.com/napcs/node-livereload)
- **Testing**: [AVA](https://github.com/avajs/ava)
- **Linting**: [ESLint](https://eslint.org/) with xo-space config
- **Versioning**: [standard-version](https://github.com/conventional-changelog/standard-version) for changelog generation

## Development Setup

### Prerequisites

- Node.js >= 20.11.0
- npm or yarn

### Initial Setup

```bash
# Fork the repository on GitHub
# Clone your fork locally
git clone https://github.com/YOUR_USERNAME/docsify-cli-plus.git
cd docsify-cli-plus

# Install dependencies
npm install

# Build the project
npm run build

# Link for global testing
npm link
```

## Available Commands

### Development Commands

| Command | Description | When to Use |
|---------|-------------|-------------|
| `npm run lint` | Run ESLint on all files | Before committing to check code style |
| `npm run lint:fix` | Run ESLint with auto-fix | To automatically fix style issues |
| `npm run build` | Build the CLI with Rollup | After any code changes to lib/ |
| `npm test` | Run all tests | Before submitting PR |
| `npm link` | Create global symlink | For testing CLI locally |

### Release Commands

| Command | Description | When to Use |
|---------|-------------|-------------|
| `npm run release` | Bump version, update changelog, create tag | When releasing new version |
| `npm run prerelease` | Build before release | Automatically run before release |

### Build Process

The build process uses Rollup to bundle the CLI:

1. **Input**: `lib/cli.js`
2. **Output**: `bin/docsify` (executable)
3. **Plugins**:
   - `rollup-plugin-executable`: Makes output file executable
   - Shebang injection: `#!/usr/bin/env node`

After building, the CLI can be run via:
```bash
./bin/docsify <command>
# or if linked globally:
docsify <command>
```

## Development Workflow

### Making Changes

1. **Create a branch**:
   ```bash
   git checkout -b feat/your-feature-name
   # or
   git checkout -b fix/bug-description
   ```

2. **Make your changes** to the source code in `lib/`

3. **Build the project**:
   ```bash
   npm run build
   ```

4. **Test your changes**:
   ```bash
   # Test locally
   npm link
   docsify <command>

   # Run tests
   npm test
   ```

5. **Check code style**:
   ```bash
   npm run lint
   ```

### Testing Locally

After `npm link`, you can test the CLI from any directory:

```bash
# Test init command
cd /tmp
mkdir test-docs
cd test-docs
docsify init

# Test start command
docsify start

# Test with options
docsify start --port 4000 --open
```

## Commit Convention

We use [Conventional Commits](https://www.conventionalcommits.org/) with [commitlint](https://commitlint.js.org/) rules. This enables automatic changelog generation.

### Commit Message Format

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Types

| Type | Description | Example |
|------|-------------|---------|
| `feat` | New feature | `feat: add start command` |
| `fix` | Bug fix | `fix: resolve sidebar loading issue` |
| `docs` | Documentation changes | `docs: update README` |
| `style` | Code style (formatting) | `style: fix indentation` |
| `refactor` | Code refactoring | `refactor: simplify server setup` |
| `perf` | Performance improvements | `perf: optimize file watching` |
| `test` | Adding/updating tests | `test: add e2e tests for start` |
| `chore` | Build/tooling changes | `chore: update dependencies` |
| `ci` | CI/CD changes | `ci: add GitHub Actions` |

### Scopes (Optional)

- `commands` - CLI commands (init, serve, start, generate)
- `template` - HTML templates
- `util` - Utility functions
- `deps` - Dependencies
- `docs` - Documentation

### Examples

```bash
# Feature commit
feat(commands): add start command for zero-config serving

Add new 'start' command that provides zero-configuration document server
with auto-generated sidebar and default index.html template.

# Bug fix commit
fix(commands): resolve virtual sidebar loading with user index.html

Fix history fallback to properly serve virtual _sidebar.md when user
provides custom index.html.

# Documentation commit
docs: add Chinese translation for README

# Dependency update
chore(deps): bump docsify to v5.0.0-rc.4
```

### Breaking Changes

For breaking changes, add `BREAKING CHANGE:` in the footer:

```
feat(commands): change default port to 8080

BREAKING CHANGE: Default port changed from 3000 to 8080
```

## Code Style

We use ESLint with `xo-space` configuration. Key rules:

- **Indentation**: 2 spaces
- **Quotes**: Single quotes
- **Semicolons**: Required
- **Line endings**: Unix (LF)
- **Max line length**: 100 characters

### Automatic Formatting

```bash
# Fix all auto-fixable issues
npm run lint:fix
```

### Pre-commit Hooks

Husky runs lint-staged before each commit:
- Automatically runs `eslint --fix` on staged `.js` files
- Adds fixed files back to the commit

## Testing

### Running Tests

```bash
# Run all tests
npm test

# Run with verbose output
npm test -- --verbose

# Run specific test file
npx ava e2e/cli.test.js
```

### Test Structure

Tests are in `e2e/cli.test.js` using AVA:
- **Unit tests**: Test individual functions
- **Integration tests**: Test CLI commands end-to-end
- **Snapshots**: Test output consistency

### Adding Tests

When adding new features, add corresponding tests:

```javascript
test('start command serves default index', async t => {
  const result = await execa('node', ['bin/docsify', 'start', '--help'])
  t.true(result.stdout.includes('Start a server'))
})
```

## Submitting Changes

### Pull Request Process

1. **Ensure tests pass**:
   ```bash
   npm test
   ```

2. **Ensure linting passes**:
   ```bash
   npm run lint
   ```

3. **Update documentation** if needed:
   - README.md for user-facing changes
   - CONTRIBUTING.md for dev process changes

4. **Create Pull Request**:
   - Use a clear title following commit conventions
   - Describe what changed and why
   - Reference any related issues

### PR Title Format

```
<type>(<scope>): <description>

Example:
feat(commands): add start command with virtual sidebar
```

### Review Checklist

Before submitting PR, verify:

- [ ] Code follows style guidelines (`npm run lint`)
- [ ] Tests pass (`npm test`)
- [ ] Build succeeds (`npm run build`)
- [ ] Documentation updated
- [ ] Commit messages follow convention
- [ ] Changes are backward compatible (or marked as breaking)
